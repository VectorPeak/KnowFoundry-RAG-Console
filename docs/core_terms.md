# 核心术语速查表

这张表用于第一遍学习时快速对齐概念。建议先理解每个术语在项目里的职责，再回到对应讲义和代码细看实现。

## 一、RAG 主链路术语

| 术语 | 一句话解释 | 项目里的体现 | 学习重点 |
|------|------------|--------------|----------|
| RAG | 先从知识库检索证据，再让大模型基于证据回答。 | `qa_core/services/qa_service.py`、`qa_core/pipeline/` | 不让模型凭空回答，核心是“证据优先”。 |
| 意图识别 | 判断用户问题属于课程咨询、政策解释、表格查询、费用承诺等哪类。 | `qa_core/intent/` | 意图会影响检索参数、Prompt 模板和风控规则。 |
| 查询改写 | 把口语化问题整理成更适合检索的表达。 | `generate_query_variants()` 所在的查询改写逻辑 | 改写不是替用户回答，而是提高召回。 |
| 查询变体 | 针对同一个问题生成多个检索表达。 | 多路检索前的 query variants | 解决“用户说法”和“文档写法”不一致的问题。 |
| Dense 检索 | 用向量相似度找语义接近的内容。 | BGE-M3 embedding + Milvus dense vector | 适合语义相近但关键词不同的问题。 |
| Sparse / BM25 | 用词项匹配找关键词重合的内容。 | Milvus 2.6 Hybrid Search 中的 sparse 通道 | 适合编号、术语、制度条款、表格字段等精确匹配。 |
| Hybrid Search | 同时使用 dense 和 sparse，再按权重融合。 | Milvus 混合检索 | 兼顾语义召回和关键词召回。 |
| Reranker | 对初步召回结果重新排序。 | CrossEncoder reranker | 提升最终进入上下文的证据质量。 |
| Context Builder | 把召回文档整理成可给 LLM 使用的上下文。 | 上下文构建模块 | 控制来源编号、内容长度、表格优先级和引用格式。 |
| Prompt Profile | 针对不同问题类别使用不同提示词模板。 | Prompt profile 配置 | 费用、就业承诺、政策、表格类问题不能用同一套回答口径。 |
| 结构化输出 | 要求回答有固定格式，比如结论、依据、风险、来源。 | Prompt 模板和输出约束 | 降低答案随意性，便于面试讲“可控生成”。 |

## 二、入库与资料治理术语

| 术语 | 一句话解释 | 项目里的体现 | 学习重点 |
|------|------------|--------------|----------|
| Loader 注册表 | 按文件类型选择成熟的 LangChain Loader。 | `qa_core/indexing/` | 避免在业务代码里堆 `if suffix == ...`。 |
| 文档标准化 | 把不同来源的文件统一成 LangChain `Document`。 | 入库预处理链路 | 后续切分、向量化、检索都基于统一结构。 |
| Parent-Child Chunking | 父块保留完整上下文，子块用于精细检索。 | 第 14 讲和切分模块 | 解决 chunk 太大召回不准、太小上下文断裂的问题。 |
| FAQ 分支 | 标准问答单独入库，命中高置信问题时可快速回答。 | FAQ CSV 入库和 FAQ 检索分支 | FAQ 是高确定性知识，不应完全混在正文里。 |
| Table Row | Excel/CSV 每一行转成一个独立 `Document`。 | `content_type=table_row` | 表格问题要保留表头、工作表、行号和单元格键值。 |
| `prefer_table` | 判断当前问题是否更像表格、清单、台账查询。 | 检索计划中的表格偏好 | 命中后提高表格行召回优先级，并禁止相似 FAQ 抢答。 |
| OCR | 从图片或扫描件中识别文字。 | 复杂图文资料入库治理 | OCR 只解决“看见文字”，不等于理解图片含义。 |
| VLM | 视觉语言模型，用于理解图片、图表、截图里的语义。 | 后续复杂资料治理规划 | VLM 适合解释图像区域、表格截图、流程图含义。 |
| 图文混排 | 同一份资料里同时有正文、图片、截图、表格。 | 第 14 讲复杂图文资料治理 | 属于多模态入库治理，不等于在线多模态聊天。 |
| `image_text_block` | 将 OCR/VLM 得到的图片文字块作为可检索证据入库。 | 复杂图片资料的目标数据结构 | 图片证据要记录页码、区域、识别方式和置信度。 |
| IndexManifest | 记录文件指纹、chunk 数量和入库时间的增量清单。 | `.index_manifest/` | 避免每次改一个文件都全量重建知识库。 |
| 知识库版本 | 每次入库生成候选版本，评测通过后再激活。 | `KnowledgeBaseVersion` | 支持回滚、对比、灰度和版本隔离。 |

## 三、多场景与数据隔离术语

| 术语 | 一句话解释 | 项目里的体现 | 学习重点 |
|------|------------|--------------|----------|
| 场景包 | 一套独立的业务资料、Prompt、评测集和演示问题。 | `scenarios/` | 多场景教学不是堆功能，而是复用同一套 RAG 能力。 |
| DataScope | 一次检索允许访问的数据范围。 | `qa_core/governance/data_scope.py` | 防止跨租户、跨数据集、跨权限误召回。 |
| `tenant_id` | 租户标识。 | 数据隔离 metadata | 多企业或多客户共用系统时必须隔离。 |
| `dataset_id` | 数据集标识。 | 数据隔离 metadata | 同一租户下也可能有多个知识库。 |
| `visibility` | 数据可见级别，比如 public、internal、private。 | 数据隔离 metadata | 决定资料是否对当前用户可见。 |
| `allowed_roles` | 允许访问资料的角色集合。 | 数据隔离 metadata | 处理 HR、法务、财务等角色权限差异。 |
| `source` | 资料来源类别，比如 policy、contract、delivery。 | 场景包 metadata | 让检索和回答能区分制度、合同、验收、理赔等资料。 |
| 业务分类 | 场景内部的二级分类。 | `business_category` | 用于更精细地调节检索权重和 Prompt。 |

## 四、质量、评测与观测术语

| 术语 | 一句话解释 | 项目里的体现 | 学习重点 |
|------|------------|--------------|----------|
| 入库质量检查 | 入库后先检查 chunk、metadata、低质量文本和异常文件。 | 质量报告脚本和重建流程 | 不合格知识不能直接激活。 |
| RAG 回归验收 | 用标准问题集验证召回和回答质量。 | `eval_sets/`、评测脚本 | 防止改代码后 RAG 主链路悄悄退化。 |
| Recall@K | 前 K 个召回结果里是否包含正确证据。 | 检索评测指标 | 衡量“有没有把正确资料找出来”。 |
| MRR | 正确证据排名越靠前，分数越高。 | 检索评测指标 | 衡量“正确资料排得够不够靠前”。 |
| 关键词覆盖 | 答案是否覆盖评测问题要求的关键事实。 | 答案评测指标 | 比单纯看语义相似更适合教学项目。 |
| 场景隔离率 | 检索结果是否来自正确场景和数据域。 | 多场景评测 | 衡量数据隔离是否真的生效。 |
| Bad Case | 已知失败样例，比如召回错、答非所问、越权召回。 | Bad Case 记录和回归测试 | 失败样例要沉淀成评测资产。 |
| LangSmith Trace | 记录一次问答经过了哪些阶段、耗时、命中哪些证据。 | `qa_core/observability/langsmith_adapter.py` | 排查“为什么没答好”的第一现场。 |
| 历史报告复盘 | 查看入库报告、评测报告、容量报告等历史结果。 | `reports/` | 方便复盘、对比版本和面试展示工程闭环。 |
| 容量评估 | 估算文档量、chunk 数、向量数量、检索耗时和存储压力。 | 容量评估脚本 | 判断当前架构能否支撑更大知识库。 |
| 企业 overlay | 用更贴近企业真实资料的字段、噪声和术语覆盖样例数据。 | 场景资料增强脚本 | 让教学数据更像业务现场，但不扩散非核心功能。 |

## 五、容易混淆的概念

| 容易混淆 | 正确区分 |
|----------|----------|
| OCR 和 VLM | OCR 负责识别图片里的文字，VLM 负责理解图片里的语义。 |
| 图文混排和多模态在线问答 | 本项目一期重点是“多模态入库治理”，不是让用户上传图片后实时对话。 |
| `table_row` 和普通 chunk | 表格行不递归切分，重点保留行号、表头和单元格键值；普通 chunk 主要保留正文语义。 |
| RAG 回归验收和单元测试 | 单元测试验证代码逻辑，RAG 回归验收验证知识召回和答案质量。 |
| Rerank 和 Query Rewrite | Query Rewrite 发生在检索前，Rerank 发生在召回后。 |
| 场景包和新功能 | 场景包主要复用同一套 RAG 能力，不应该把每个场景都做成一套新系统。 |
| 企业 overlay 和真实生产接入 | overlay 是教学数据增强，真实生产接入还需要权限、审计、脱敏、SLA 和合规评审。 |

## 六、面试表达模板

介绍任何一个术语时，可以按这四句话组织：

1. 它解决什么问题。
2. 项目里放在哪个链路。
3. 为什么不用更简单的方案。
4. 当前边界是什么，后续如何升级。

示例：

```text
Excel/CSV 入库解决的是表格类资料按普通文本切分后行列关系丢失的问题。
项目里把每一行转成 LangChain Document，并记录 sheet_name、row_number、content_type=table_row。
这样检索时可以识别表格问题并启用 prefer_table，避免相似 FAQ 抢答。
一期只支持规范二维表，复杂合并单元格、截图表格和图表说明会进入后续 OCR/VLM 治理链路。
```
