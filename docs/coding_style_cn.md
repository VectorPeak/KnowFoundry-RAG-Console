# 中国团队可读性代码风格约定

本文档约束当前项目后端代码的阅读风格。目标不是把代码写成“中文变量名”，而是让国内团队、学习者和面试讲解时更容易理解主链路。

## 1. 命名规则

- 包名和文件名使用英文小写下划线，表达职责，例如 `pipeline/steps.py`、`quality/ingestion.py`。
- 类名和函数名继续使用英文，避免中文标识符影响编辑器、类型检查和跨团队协作。
- 变量名不追求缩写，优先使用有业务含义的名称，例如 `prepared`、`context`、`faq_result`。
- 不使用炫技式简称，例如 `ctx`、`res`、`tmp`，除非作用域非常小。

## 2. 注释规则

- 注释使用中文，重点解释“为什么这样做”和“适用场景”。
- 不写无意义注释，例如“调用函数”“返回结果”。
- 对 RAG 链路中的关键取舍必须说明原因，例如 FAQ 直出阈值、知识库版本过滤、查询变体、重排。
- 长函数拆分后，每个阶段保留 1-2 行中文说明，让读者知道当前阶段在业务上做什么。

## 3. 模块边界

- `application` 只做服务门面，不写检索细节。
- `pipeline` 只做 RAG 编排和步骤拆分，不直接写 Milvus schema。
- `retrieval` 只处理召回、重排、过滤表达式。
- `indexing` 只处理加载、清洗、切分、入库。
- `quality` 只处理入库质量、低质量 chunk、FAQ 和冲突检测。
- `governance` 只处理版本和数据隔离。
- `memory` 只处理历史和反馈。
- `observability` 只处理 trace、评测报告和统计。

## 4. 主流程写法

主流程要像流程单，而不是把所有细节堆在一个函数里。

推荐结构：

```python
# 第一阶段：识别意图，并准备检索参数。
prepared = prepare_retrieval(context)

# 第二阶段：处理无需 RAG 的直接答案。
...

# 第三阶段：先查 FAQ。
...

# 第四阶段：再查正文资料。
...

# 第五阶段：流式生成答案。
...

# 第六阶段：写历史、写 trace、发送 end 事件。
...
```

不推荐结构：

```python
def stream_query(...):
    # 在一个函数里同时做场景解析、历史读取、意图识别、FAQ 检索、
    # 文档检索、Prompt 拼接、LLM 调用、异常记录和 trace 收尾。
```

## 5. 导入规则

- 不从包入口做隐式导入，例如不要写 `from qa_core.indexing import ingest_directory`。
- 必须从具体子模块导入，例如 `from qa_core.indexing.service import ingest_directory`。
- 这样读代码时可以直接知道函数来自哪一层，也避免包入口为了“方便”变成新的转发层。

## 6. 异常处理

- 异常处理必须说明原因和影响范围。
- 在线问答链路优先返回可展示事件，不把内部异常直接抛给前端。
- 入库、检索、重排、Milvus、MySQL、LLM、本地模型等主链路依赖失败时，不允许静默改走低配路径。
- 只有“无召回上下文”这类业务结果可以返回“信息不足，无法确认”；这不是技术降级。

## 7. 教学项目的取舍

这个项目是多场景 RAG 教学项目，一期目标是把 RAG 基座做扎实，不追求过度抽象。

- 可以为了教学可读性多写一些中文注释。
- 可以保留业务阶段函数，避免初学者直接面对复杂链路。
- 不为了“设计模式完整”而增加接口层、抽象基类和泛型封装。
- 二期 Agent 应新建 `qa_core/agent`，不要把 Agent 规划逻辑塞进一期 RAG 主流程。
