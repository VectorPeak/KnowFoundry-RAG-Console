一、向量索引的本质¶

1.1 为什么需要索引¶

索引的本质：用额外的存储空间和构建时间，换取查询时的大幅加速。类比： - 无索引 = 在未排序的书架上逐本翻找 - 有索引 = 先查图书馆目录卡片，按索书号直接走到对应书架

1.2 索引在什么时候构建¶

sequenceDiagram participant User as 用户/脚本 participant Milvus as Milvus participant Disk as 磁盘 User->>Milvus: 1. create_collection(schema) Note over Milvus: Collection 创建完成
但此时没有索引 User->>Milvus: 2. insert(data) Note over Milvus: 数据进入 Growing Segment
此时搜索走暴力搜索 User->>Milvus: 3. create_index(field, index_type, params) Milvus->>Milvus: 后台异步构建索引 Note over Milvus: 构建时间取决于数据量和参数
几秒到几十分钟不等 Milvus->>Disk: 索引文件写入 MinIO/S3 User->>Milvus: 4. load_collection() Note over Milvus: 将索引加载到内存
此后搜索走索引加速 User->>Milvus: 5. search() Note over Milvus: ✅ 毫秒级响应

关键点： - 创建 Collection 时不会自动建索引——必须显式调用 create_index() - 索引构建是异步的——调用 create_index() 后立即返回，Milvus 在后台构建 - 必须先 load_collection() 将索引加载到内存，才能使用索引加速搜索

IVF_FLAT 的改进版：对向量做压缩，减少内存占用。

2.5 索引选型决策树¶

这张决策树覆盖了从原型到生产的所有数据规模，每条分支的决策逻辑如下：

分支一：数据量 < 1 万 → FLAT

FLAT 索引不做任何压缩或近似——检索时逐条计算输入向量与全部 1 万条向量的距离，然后排序返回 Top-K。1 万条向量的暴力计算在现代 CPU 上大约耗时 5-15ms，完全可接受。优势是 100% 精度（不会漏掉任何真正最近的结果），适合原型验证和小规模场景。本项目的单个场景 FAQ Collection 通常只有几十到几百条，在这个数量级上 FLAT 和 HNSW 的延迟差异几乎不可感知。

分支二：1 万 - 100 万 + 内存充足 → HNSW（本项目选择）

HNSW（Hierarchical Navigable Small World）是当前综合表现最好的近似最近邻索引。它预先构建一张多层"高速公路图"——上层节点少、跳得远，下层节点密、查得准。检索时从上层入口出发，逐层下降到精确层，每次只检查少量候选节点。

本项目的文档 Collection（每个场景几千到几万条 chunk，加上多场景和多版本，总规模在 10 万-50 万条）非常适合 HNSW——内存需求约 1-2 GB（完全在开发机构内存内），检索延迟稳定在 20-50ms，召回率 97-99%。

分支三：1 万 - 100 万 + 内存紧张 → IVF_FLAT

IVF_FLAT 用 K-means 聚类把向量空间分成 N 个桶（nlist 参数）。检索时只搜索最近的若干个桶，而不是全量扫描。内存占用比 HNSW 小（不需要存图结构），但精度也略低——如果查询向量落在两个桶的边界附近，可能会漏掉相邻桶中的真正近邻。

分支四：> 100 万 → IVF_PQ 或 DiskANN

百万级以上向量时，即使是 HNSW 的内存开销（约 2-5 GB/百万条）也可能超出预算。IVF_PQ（Product Quantization）将高维向量压缩为短编码（如 1024 维 → 64 字节），内存减少 10-20 倍，代价是精度降至 90-95%。DiskANN 则把部分索引卸载到 SSD 上，用磁盘换内存。

为什么本项目选择 HNSW？ 数据量在 10 万-50 万区间，开发机内存（16-32 GB）完全足够，不需要压缩。HNSW 在这个规模下精度最高（97-99%）、速度最快（无聚类边界问题），是目前 RAG 系统最主流的索引选型。

from pymilvus import (
    Collection, CollectionSchema, FieldSchema, DataType,
)

# 第一步：定义字段
pk_field = FieldSchema(
    name="pk",              # 主键字段名
    dtype=DataType.VARCHAR, # 字符串类型
    is_primary=True,        # 标记为主键
    max_length=128,         # 最大 128 字符
)

text_field = FieldSchema(
    name="text",            # 文本字段（被 Embedding 和 BM25 使用）
    dtype=DataType.VARCHAR,
    max_length=65535,       # 支持长文本
)

dense_field = FieldSchema(
    name="dense",           # Dense 向量字段
    dtype=DataType.FLOAT_VECTOR,
    dim=1024,               # BGE-M3 是 1024 维
)

sparse_field = FieldSchema(
    name="sparse",          # Sparse 向量字段（BM25）
    dtype=DataType.SPARSE_FLOAT_VECTOR,
)

source_field = FieldSchema(
    name="source",          # 业务分类（标量过滤用）
    dtype=DataType.VARCHAR,
    max_length=64,
)

kb_version_field = FieldSchema(
    name="kb_version",      # 知识库版本（标量过滤用）
    dtype=DataType.VARCHAR,
    max_length=128,
)

# 第二步：创建 Schema
schema = CollectionSchema(
    fields=[pk_field, text_field, dense_field, sparse_field,
            source_field, kb_version_field],
    description="教学用 FAQ 集合",
    enable_dynamic_field=True,  # 允许 metadata 的额外字段
)

# 第三步：创建 Collection
collection = Collection(
    name="demo_collection",
    schema=schema,
    consistency_level="Session",  # 会话一致性
)

    data=query_vector,          # 查询向量
    anns_field="dense",         # 在哪个向量字段上搜索
    param=search_params,
    limit=5,                    # 返回 Top-5

# 处理结果
for i, hits in enumerate(results):
    print(f"查询 {i}:")
    for hit in hits:
        print(f"  id={hit.id}, distance={hit.distance:.4f}")
        print(f"  text={hit.entity.get('text')[:50]}...")
        print(f"  source={hit.entity.get('source')}")

5.1 索引构建耗时估算¶

5.2 本项目为什么感觉不到索引的存在¶

 1

10
本项目的实际流程：
  1. 脚本 rebuild_kb_version.py 调用 store.add_documents()
  2. LangChain Milvus 自动完成 insert → flush → 后台构建索引
  3. 入库完成时，索引可能还没构建完
  4. 但数据量小（<10万），Growing Segment 暴力搜索也不慢（~10ms）
  5. 几秒后索引构建完成，后续搜索自动走 HNSW 索引加速

→ 学生体验：入库后立刻就能搜索，感觉不到索引的存在
→ 实际原因：数据量小 + Growing Segment 暴力搜索兜底
→ 生产环境：必须等索引构建完成再开放搜索，否则前几分钟用户体验很差

本讲小结¶

  • 索引的本质：用空间和构建时间换查询速度。未建索引时走暴力搜索，建了索引走 ANN 近似搜索
  • 四种主流索引：FLAT（100% 精度/最慢）、IVF_FLAT（聚类加速）、IVF_PQ（压缩+聚类）、HNSW（图搜索/本项目使用）
  • 基本操作六步走：连接 → 创建 Collection → 创建索引 → 加载 → 插入 → 搜索
  • LangChain 自动帮我们做了：初始化时自动创建 Collection + Schema + Index + Load；add_documents 时自动 Embedding + BM25 + Insert；search 时自动 Embedding + 混合融合
  • 本项目不需要调参：十万级数据 + HNSW 默认参数 = 毫秒级搜索，足够教学和演示
