第13讲 笔记：数据隔离与多租户设计¶

学习目标¶

  • 能说出 DataScope 四个隔离维度的含义和过滤实现方式
  • 能解释可见性层级（public / internal / private）的包含关系
  • 能完整描述维护一个既有业务场景需要的全部文件和步骤

核心知识点¶

  1. DataScope 四维隔离：tenant_id（租户）→ dataset_id（数据集）→ visibility（可见性层级）→ allowed_roles（角色数组）
  2. 可见性层级是包含关系：public 包含 internal 和 restricted 内容。过滤表达式为 visibility in ["public", "internal"]
  3. 所有隔离字段在入库时写入 metadata，检索时拼入 Milvus 过滤表达式，双重保护（白名单校验 + 安全转义）防注入
  4. 当前 8 个场景已经冻结；维护既有场景只需改 TOML、FAQ CSV 和资料文件，再执行版本重建，零 Python 代码修改
  5. 场景边界检测遍历全部已注册场景的 source_patterns，交叉命中时提示用户切换场景

关键概念¶

术语 定义
DataScope 封装一次查询的数据访问范围的数据类
可见性层级 public ⊃ internal ⊃ restricted，下层包含上层内容
场景边界 用户问题匹配到其他场景的 source 时触发的拦截机制
scenario.toml TOML 格式的场景配置，声明身份/collection/source/正则

# qa_core/governance/data_scope.py — 四维隔离数据类
@dataclass(frozen=True)
class DataScope:
    tenant_id: str = "default"
    dataset_id: str = "default"
    visibility: str = "public"
    user_role: str = "public"
    user_roles: list[str] = field(default_factory=list)

    def expr_clauses(self):
        """返回可追加到 Milvus expr 的过滤子句"""
        clauses = []
        if self.tenant_id:
            clauses.append(f'tenant_id == "{escape_expr_value(self.tenant_id)}"')
        if self.dataset_id:
            clauses.append(f'dataset_id == "{escape_expr_value(self.dataset_id)}"')
        # 可见性层级：当前级别能看到的内容
        if self.visibility:
            allowed = visibility_chain(self.visibility)
            clauses.append(f'visibility in {json.dumps(allowed)}')
        # 角色过滤：Milvus array_contains
        if self.user_roles:
            roles = json.dumps([escape_expr_value(r) for r in self.user_roles])
            clauses.append(f"array_contains_any(allowed_roles, {roles})")
        return clauses

# qa_core/governance/data_scope.py — 安全转义防注入
def escape_expr_value(value: str) -> str:
    """防止用户输入中含双引号等特殊字符破坏 Milvus 表达式"""
    return str(value).replace('"', '\\"')

练习题¶

  1. 模拟数据隔离：对同一场景，用不同 tenant_id 参数分别提问，通过 debug 接口确认检索结果是否被隔离字段正确过滤。
  2. 维护一个既有场景：按讲 14 第七部分的 7 步指南，为 engineering_project_qa 补充一条 FAQ 和一份资料，完成入库和提问，验证整个流程不需要改代码。

常见面试题¶

Q1：你们的轻量多租户方案有什么局限？

当前方案通过在 Milvus expr 中追加过滤实现隔离，数据仍存储在同一个 Collection 中。大规模场景建议升级为 Partition Key 或 Database 级物理隔离。但对于教学和演示规模，表达式过滤完全够用。

Q2：如何防止用户构造特殊输入绕过数据隔离？

两道防线：白名单校验（source_filter 必须在 valid_sources 中）+ 安全转义（双引号等特殊字符被转义）。即使前端绕过白名单，后端在 build_source_expr 中仍会校验和转义。
