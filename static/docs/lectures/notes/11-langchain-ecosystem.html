一、学习目标（可验证）¶

  • 能说出 LangChain 的三大核心抽象（Model/Chain/Agent）及在本项目中的使用策略
  • 能解释 Runnable 接口的三个核心方法（invoke/stream/batch）及各自适用场景
  • 能写出 LCEL 管道符 | 串联 prompt → model → parser 的代码，并解释本项目为何不用
  • 能区分 ChatOpenAI 三种调用模式：invoke（非流式）、stream（流式）、with_structured_output（Pydantic 约束）
  • 能说出 SystemMessage/HumanMessage/AIMessage 与 OpenAI API role 的映射关系
  • 能解释 ChatPromptTemplate + MessagesPlaceholder 如何构建带动态对话历史的 Prompt
  • 能说明 Document Loader 注册表模式的设计思路和扩展方式
  • 能解释 RecursiveCharacterTextSplitter 递归降级切分算法和父子块策略的价值

二、知识点详解¶

2.1 LangChain 核心定位与三大抽象¶

是什么：LangChain 不是模型，而是 LLM 应用开发框架。它将对话历史、文档加载、文本切分、向量存储、提示模板等环节做标准化封装。三大抽象按灵活性从低到高：

Model（模型） → Chain（链） → Agent（智能体）
统一调用接口    管道串联组件    LLM 自主决策下一步

本项目使用策略：✅ 全面使用 Model 层（ChatOpenAI 封装 DashScope）；⚠️ 部分使用 Chain 底层组件（不用管道符串联）；🔮 二期规划 Agent 层（LangGraph 动态工具调用）。

2.2 Runnable 统一接口¶

是什么：LangChain 所有组件都实现相同接口——invoke()（等完整结果）、stream()（逐 token 流式输出）、batch()（并行处理多个输入）。

为什么需要：早期不同组件调用方式不同（llm.predict()、chain.run()、retriever.get_relevant_documents()），换组件就要换 API。Runnable 统一了这一切。

model = ChatOpenAI(model="qwen-plus")

# invoke：等完整结果（用于意图识别等需要完整结果继续下一步的场景）
response = model.invoke([HumanMessage("入职流程有哪些步骤")])
# → AIMessage(content="入职流程包括：1. 提交材料...")

# stream：逐个 token 返回（用于前端实时渲染）
for chunk in model.stream([HumanMessage("入职流程有哪些步骤")]):
    print(chunk.content, end="")  # 50ms→"入", 100ms→"职", ...

# batch：并行处理多个问题（用于批量评估、测试）
responses = model.batch([
    [HumanMessage("入职流程")], [HumanMessage("报销规则")],
])  # → [AIMessage, AIMessage]

2.3 LCEL 管道符与为何本项目不用¶

LCEL（LangChain Expression Language）用 | 管道符串联 Runnable：

1
2
chain = prompt | model | parser
# 数据流：prompt.invoke(input) → model.invoke(prompt_output) → parser.invoke(model_output)

本项目为什么不用：四个原因——① RAG 有 FAQ 直出/文档RAG/信息不足/直接答案等多种分支，不适合线性管道；② 自编流程每一步都有 reason 字段可解释；③ 每阶段输入输出可单独调试；④ RAG 需要在阶段之间做动态决策（FAQ 命中直接返回、检索不足追加搜索）。

Structured Output 工作原理：LangChain 将 Pydantic 模型的 JSON Schema 注入 System Prompt → LLM 按 Schema 返回 JSON → Pydantic 自动校验。枚举值超范围直接报错，不需要手写正则。

2.5 Message 类型系统¶

LangChain 类OpenAI API role用途
SystemMessagesystem定义 AI 角色和行为边界
HumanMessageuser用户说的话
AIMessageassistantAI 的回答（多轮对话历史）

关键理解：LLM 是无状态的——每次请求必须完整发送历史消息列表。这就是多轮对话的本质：把 `[System, User1, AI1, User2, AI2, User3]` 全部发给 LLM，让它"记住"上下文。

2.6 Prompt Templates 与 MessagesPlaceholder¶

ChatPromptTemplate：用模板变量管理 System/User Prompt，变量在运行时从场景 TOML 配置注入，使同一套模板可用于全部 8 个行业场景。

MessagesPlaceholder：当消息数量不固定时（多轮对话历史），在模板中留一个"消息列表"插槽，调用时传入 [HumanMessage(...), AIMessage(...), ...] 列表自动展开。

本项目 PromptProfile 数据类：qa_core/prompts/profiles.py 用 frozen dataclass 管理多档位模板（FAQ_QUERY / KNOWLEDGE_QUERY / FOLLOW_UP），防止运行时篡改。

2.8 Document Loaders — 注册表模式¶

设计思路：按文件扩展名自动匹配加载器（PDF→PyPDFLoader, Word→Docx2txtLoader, Markdown→TextLoader, Excel→自定义表格加载器），新增格式只需加一条注册项 DocumentLoaderSpec(suffixes=(".xyz",), factory=_xyz_loader, ...)，主流程零修改。符合开闭原则——对扩展开放，对修改关闭。

📁 完整源码：qa_core/indexing/document_loaders.py

2.9 Text Splitters — 文档切分¶

RecursiveCharacterTextSplitter 递归降级切分：按优先级尝试分隔符——\n\n（段落）→ \n（换行）→ 。（句子）→ ，（短语）→ 空格 → 字符（最后手段）。优先在语义边界切分，字符截断是最后的应急手段。

父子块策略：子块（500 字符）存入 Milvus 做精确召回；父块（2000 字符）存于子块 metadata.parent_content——检索命中子块后，LLM 看到的是父块的完整上下文。精确 + 完整兼得。

📁 完整源码：qa_core/indexing/chunking.py → split_documents()

三、常见面试题¶

Q1：Runnable 统一接口解决了什么问题？

早期不同组件调用方式不同（llm.predict()、chain.run()），换组件就换 API。Runnable 统一为 invoke(完整)/stream(流式)/batch(并行)，所有组件调用方式一样，可灵活替换和组合。

Q2：with_structured_output 相比手写正则有什么优势？

手写正则三大问题：LLM 返回格式不稳定（有时中文有时英文）；字段约束无法强制执行（confidence 0-1、intent 枚举值）；每次改 schema 都要重写正则。with_structured_output 将 Pydantic Schema 注入 System Prompt，Pydantic 自动校验——枚举值超范围直接报错，schema 变更只改模型不改解析逻辑。

Q3：为什么不用 LCEL 管道符串联 RAG 流程？

四个原因：① 分支复杂（FAQ 直出/文档RAG/信息不足/直接答案）；② 可解释性（每步有 reason 字段）；③ 调试友好（每阶段输入输出可单独检查）；④ 动态决策（FAQ 命中直接返回、检索不足追加搜索）。LCEL 适合简单线性链，不适合有决策逻辑的工程链路。

Q4：注册表模式和 if/elif 分支有什么区别？

if/elif 把每种格式写死在主流程中。注册表模式把"后缀→工厂函数"映射集中管理，新增格式加一条注册项，主流程只查表不关心具体类型。对扩展开放，对修改关闭——开闭原则。

Q5：父子块切分策略的价值？

单一切块矛盾：chunk 太大→检索不精确；chunk 太小→LLM 上下文断裂。父子块解决：子块（500字）精确召回命中；父块（2000字）从 metadata.parent_content 展开给 LLM 完整上下文。精确 + 完整兼得。

Q6：ChatPromptTemplate 的 MessagesPlaceholder 解决了什么问题？

普通模板变量只能是字符串，但多轮对话历史长度不固定（可能是 2 条也可能是 20 条消息）。MessagesPlaceholder 在模板中留"消息列表"插槽，调用时传入完整 Message 列表自动展开。同一模板既能处理首轮提问（history=[]），也能处理多轮追问。

五、练习题¶

  1. 对比 invoke 和 stream：用 get_chat_model(streaming=False).invoke() 和 get_chat_model(streaming=True).stream() 分别调用同一个问题，观察返回方式的差异。
  2. 探索 structured output：在 IntentLLMDecision 中新增一个字段（如 urgency: Literal["low", "medium", "high"]），观察 LLM 返回的 JSON 是否自动包含该字段。
  3. 新增文件格式：按照注册表模式，为 .json 文件编写一个 loader 工厂函数并注册到 DOCUMENT_LOADER_SPECS，验证 load_file() 能否正确处理。
  4. chunk 大小实验：修改 child_chunk_size 为 200/500/1000 分别切分同一篇长文档，观察不同大小下检索结果的精确度和上下文完整性差异。
