第1讲 笔记：项目概述与环境搭建¶

学习目标¶

  • 能说清 RAG 的基本概念和离线/在线两条链路
  • 能启动 Docker 服务并让项目跑起来
  • 能说出 8 个业务场景的名称和定位

核心知识点¶

  1. RAG = 检索增强生成：先从知识库检索相关文档，再把文档和问题一起发给 LLM，让 LLM 基于资料回答
  2. 两条链路分工明确：离线链路负责入库（加载→切分→向量化→存储），在线链路负责问答（意图→检索→重排→生成）
  3. 技术栈主线：FastAPI + LangChain + Milvus + BGE-M3 + DashScope，前端是原生静态页面
  4. 8 个场景共享同一套核心引擎，靠 scenario.toml 配置切换，不是 8 套代码
  5. 启动时执行前置校验，Milvus/MySQL/LLM/模型/场景任一缺失直接启动失败，不允许降级

关键概念¶

术语 定义
RAG Retrieval-Augmented Generation，检索增强生成
Embedding 将文本转为固定长度向量的技术，语义相近的文本向量距离也近
Milvus 开源向量数据库，支持 Dense + Sparse 混合检索
BGE-M3 北京智源研究院开发的中文 Embedding 模型，1024 维输出
Preflight Check 启动前校验所有核心依赖是否可用，失败则拒绝启动

PLACEHOLDER_VALUES = {"", "replace-with-real-key", "changeme"}
def _is_placeholder(value):
    return str(value or "").strip().lower() in PLACEHOLDER_VALUES

def _require_tcp(name, host, port, timeout=3.0):
    sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    sock.settimeout(timeout)
    result = sock.connect_ex((host, port))
    sock.close()
    if result != 0:
        raise RuntimeError(f"{name} ({host}:{port}) 不可连接")

练习题¶

  1. 维护一个既有场景：选择 engineering_project_qa 或 insurance_claims，阅读 scenario.toml 中的 source 白名单、关键词规则和示例问题，理解页面下拉框和 source 推断为什么不需要改 Python 代码。
  2. 验证前置校验：故意把 .env 中的 DASHSCOPE_API_KEY 改成一个无效值，重启服务，观察启动失败日志中是否明确指出问题。
  3. 跨场景切换提问：在页面切换到 cross_border_risk，提问「新人入职流程怎么走」，观察是否触发场景边界提示。

常见面试题¶

Q1：你们项目的 RAG 系统是怎么工作的？

用户提问后先做意图识别，然后 FAQ + 文档分层混合检索（Milvus Dense + BM25 Sparse），BGE Reranker 重排后构建上下文，按 Prompt Profile 调用 LLM 流式生成答案。每次问答都写入 trace 用于复盘。

Q2：为什么生产环境选 Milvus 而不是 pgvector？

pgvector 百万级以上向量检索性能不够，且不支持混合检索。Milvus 原生支持 Dense + Sparse Hybrid Search，十亿级向量毫秒级响应，同时支持丰富的标量过滤表达式。
