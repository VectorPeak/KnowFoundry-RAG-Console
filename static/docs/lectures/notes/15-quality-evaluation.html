第15讲 笔记：RAG 回归验收与入库质量¶

学习目标¶

  • 能说出三层保障体系（入库质量 / 检索评测 / 性能基线）各自检查什么
  • 能手算 Recall@K 和 MRR，能用具体例子解释计算过程
  • 能描述 Bad Case 如何从 LangSmith trace/annotation 沉淀为回归样本

核心知识点¶

  1. 三层保障：入库质量（资料健康）→ 检索评测（策略有效）→ 性能基线（响应合理），三层都通过才能激活版本
  2. Recall@K 衡量期望关键词在 Top-K 召回文档中的覆盖率。手算：5 个期望关键词中有 4 个出现在 Top-5 文档中 → Recall@5 = 0.8
  3. MRR 衡量第一个相关文档的排名倒数均值。手算：三查询分别排第 1/3/1 位 → MRR = (1+0.333+1)/3 ≈ 0.778
  4. 分组验收防止局部退化被全局均值掩盖：按场景、source、hit_type 分别设验收阈值
  5. Bad Case 沉淀：LangSmith Trace → Annotation → Dataset → Evaluation → 下次评测必须通过

关键概念¶

术语 定义
Recall@K Top-K 文档中覆盖期望关键词的比例
MRR 第一个相关文档排名的倒数平均值
分组验收 按场景/source/hit_type 分别检查，防止局部退化
Bad Case Promotion 将 LangSmith 标注后的问题样本沉淀为回归评测集的一部分

# scripts/eval_common.py — 评测指标计算（核心逻辑）
class RAGEvaluationMetrics:
    def compute(self, test_cases, actual_results):
        # Recall@K：期望关键词在召回文档中的覆盖率
        metrics["recall_at_k"] = sum(
            self._keyword_recall(tc, result)
            for tc, result in zip(test_cases, actual_results)
        ) / len(test_cases)

        # MRR：第一个相关文档排名的倒数平均值
        metrics["mrr"] = sum(
            1.0 / self._first_relevant_rank(tc, result)
            for tc, result in zip(test_cases, actual_results)
        ) / len(test_cases)

        # Source 推断准确率
        metrics["source_inference_accuracy"] = sum(
            1.0 if tc["expected_source"] == result.get("source_filter") else 0.0
            for tc, result in zip(test_cases, actual_results)
        ) / len(test_cases)
        return metrics

# scripts/check_evaluation_gate.py — 分组验收（防止局部退化被全局掩盖）
def check_evaluation_gate(report):
    failures = []
    # 全局验收
    if report["metrics"]["recall_at_k"] < 0.85:
        failures.append(f"全局 Recall@K 不达标")
    # 按场景分组 ← 防止某个场景退化被全局均值掩盖
    for scenario_id, m in report["by_scenario"].items():
        if m["recall_at_k"] < 0.80:
            failures.append(f"场景 {scenario_id} Recall@K {m['recall_at_k']} < 0.80")
    return len(failures) == 0, failures

练习题¶

  1. 手算评测指标：取 eval_sets/multi_scenario_smoke.json 中的一条样本，手动跑 debug_retrieval，用返回的 faq_sources/doc_sources 手算该样本的 Recall@K。
  2. 沉淀 Bad Case：从 LangSmith trace 中选择异常记录，标注后加入 dataset，判断是否需要进入回归评测。

常见面试题¶

Q1：为什么回归验收要分组检查而不是只看全局平均？

全局平均会掩盖局部退化。8 个场景中 7 个 Recall@K=1.0、1 个降到 0.5，全局平均仍有 0.937 看起来没问题。分组验收要求每个场景 ≥0.8、每个 source ≥0.75，退化场景立刻暴露。

Q2：评测集怎么设计才有效？

评测样本必须包含 expected_source、expected_keywords（至少 4-6 个具体词）、expected_hit_type、min_expected_sources。不能只丢一个问题期望召回好，需要精确到每个维度的预期行为。
