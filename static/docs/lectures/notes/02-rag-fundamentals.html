第2讲 笔记：RAG 核心概念深入¶

一、学习目标（可验证）¶

  • 能用自己的话解释 Embedding 是什么，为什么"语义相近 = 向量距离近"
  • 能手算余弦相似度，说出余弦相似度、欧几里得距离、内积三种计算方式的区别
  • 能解释 Dense 检索（语义）和 Sparse 检索（关键词 BM25）各自的优势和局限
  • 能说明 Hybrid Search（混合检索）为什么要把 Dense + Sparse 结合起来
  • 能画出 Bi-Encoder（粗排）→ CrossEncoder（精排）的两阶段检索链路
  • 能解释 Reranker 为什么比向量相似度更准，以及为什么不能对所有文档都用 Reranker
  • 能回答面试题：Dense 检索和 Sparse 检索的区别是什么？为什么要重排？

二、知识点详解¶

2.1 Embedding：文本 → 向量¶

是什么：Embedding 将非结构化文本转换成固定长度的浮点数向量。本项目使用 BGE-M3 模型，将任意长度的中英文文本转成 1024 维向量——语义相近的文本在向量空间中距离更近。

为什么需要：计算机只能做数值计算。传统 TF-IDF 词频统计无法区分"我很开心"和"我非常高兴"是同一意思，也无法区分"银行利率很高"和"河边的银行很美"中"银行"的不同含义。Embedding 基于 Transformer 架构做上下文语义理解，解决了这个问题。

import numpy as np

# 两个语义相近的句子的向量（模拟 BGE-M3 输出）
A = np.array([0.5, 0.3, 0.8, 0.1, 0.6])  # "入职需要什么材料"
B = np.array([0.48, 0.32, 0.79, 0.09, 0.58])  # "入职要准备哪些文件"

cosine_sim = np.dot(A, B) / (np.linalg.norm(A) * np.linalg.norm(B))
# → 0.95（很高 → 语义相近）

C = np.array([-0.3, 0.7, -0.2, 0.5, -0.1])  # "今天天气怎么样"
cosine_sim_ac = np.dot(A, C) / (np.linalg.norm(A) * np.linalg.norm(C))
# → 0.1（很低 → 语义无关）

2.2 三种向量相似度计算方式¶

方法公式取值范围特点
余弦相似度dot(A,B) / (‖A‖×‖B‖)[-1, 1]只看方向不看长度，最常用于文本
欧几里得距离√Σ(Aᵢ-Bᵢ)²[0, +∞)L2 归一化后等价于余弦
内积（点积）Σ(Aᵢ×Bᵢ)(-∞, +∞)Milvus 默认，计算最快

关键理解：BGE-M3 输出时已做 L2 归一化（normalize_embeddings=True），使所有向量长度=1。此时内积数学上等价于余弦相似度，但少一次开方运算，性能更好。

2.4 向量数据库：为什么是 Milvus¶

对比维度pgvectorMilvus 2.6
百万级向量检索秒级毫秒级
索引类型IVFFlat, HNSWHNSW, IVF, DiskANN 等 11 种
混合检索 (Dense+Sparse)不支持原生支持
分布式需额外配置原生支持

Milvus 核心概念：Collection（数据库表）→ Field（主键/向量/文本/标量列）→ Index（HNSW 加速检索）→ Partition（物理分片）。部署架构：etcd（元数据） + MinIO（索引文件） + Milvus（核心引擎）三个容器。

2.5 Dense 检索 vs Sparse 检索（核心）¶

Dense 检索：Embedding 模型把问题和文档都转成向量，算余弦相似度。理解语义，能识别同义词和改写。Sparse 检索：BM25 算法做关键词精确匹配。不理解语义，但精确术语命中率极高。

举例：用户问"我怎么修改自己的登录密码"——Dense 能将"修改密码"和"忘记密码→自助重置"语义匹配（0.91），而 Sparse 只在"密码"关键词上统计词频。另一场景"API 密钥在哪里生成"——Dense 可能把 API 相关文档全召回（难以精确定位），而 Sparse 精确命中"API""密钥""生成"。

Hybrid Search = Dense + Sparse：Milvus 一次查询同时走两路，weighted ranker 加权融合（本项目 Dense 权重 0.55，Sparse 权重 0.45）。

2.6 Reranker（重排器）：从粗排到精排¶

是什么：两阶段检索——阶段一用 Bi-Encoder（向量距离）快速圈定 Top 30 候选；阶段二用 CrossEncoder（[问题+文档]拼接编码，交叉注意力打分）精细重排取 Top 5。

为什么需要：Bi-Encoder 问题和文档分别编码，没有交互，精度有限。典型问题："离职需要什么材料"在搜"入职需要什么材料"时向量相似度也可能很高（0.78）。CrossEncoder 将"问题+文档"拼接编码，能区分"入职"和"离职"，将误召回排到最后。

# qa_core/retrieval/ranking.py — CrossEncoder 重排
def rerank_hits(query, hits, *, reranker, top_n):
    if not hits: return []
    # 构建 [query, document] 对，CrossEncoder 联合编码
    pairs = [(query, hit.document.page_content) for hit in hits]
    scores = reranker.predict(pairs)  # CrossEncoder 精细打分
    reranked = [
        RetrievalHit(document=hit.document, score=float(score))
        for hit, score in sorted(zip(hits, scores), key=lambda x: float(x[1]), reverse=True)
    ]
    return reranked[:top_n]

2.7 完整检索链路¶

用户问题 → BGE-M3 Embedding → Milvus 混合检索(Dense+Sparse) → 去重合并(Top 30) → BGE Reranker 重排 → Top 5 进入 LLM 上下文

三、常见面试题¶

Q1：Dense 检索和 Sparse 检索的核心区别？各自适合什么场景？

Dense（语义检索）：Embedding 模型向量化后计算余弦相似度。理解语义，能识别同义词和改写。适合自然语言问答、长文本语义匹配。局限是对精确术语区分能力弱。

Sparse（关键词检索）：BM25 词频统计做精确匹配。不理解语义但精确术语命中率极高。适合编号查询、法规条文搜索。局限是无法识别同义词。实际项目中二者互补，混合检索取长补短。

Q2：为什么要用 Reranker？为什么不直接用 Milvus 的向量分数？

Embedding 检索是 Bi-Encoder（双塔架构）：问题和文档分别编码，没有交叉注意力，精度有限。典型问题：语义相近但答非所问的文档被排到前面（"离职需要什么材料"在检索"入职需要什么材料"时向量相似度可能也很高）。

Reranker 使用 CrossEncoder：将问题和文档拼接后一起编码，通过交叉注意力获得精确相关性判断，能区分"入职"和"离职"。代价是计算量大，所以只对 Top 20~50 条候选做，而不是对所有文档做。

Q3：BGE-M3 的核心特性是什么？为什么选它？

多语言（中英+100+语言）、多粒度（短句到8192 token长文档）、多功能（同时输出 Dense + Sparse 向量）。本地运行不调外部 API，数据安全。一个模型覆盖两种检索方式。

Q4：余弦相似度和内积有什么区别？为什么 Milvus 默认用内积？

余弦相似度 = 内积 / (‖A‖×‖B‖)，对向量长度做了归一化。内积同时受方向和长度影响。但 BGE-M3 输出时已做 L2 归一化（normalize_embeddings=True），‖A‖=‖B‖=1，此时内积等价于余弦相似度。内积少一次开方运算，性能更好。

Q5：Hybrid Search 中 Dense 和 Sparse 的权重怎么定？

本项目 Dense 0.55，Sparse 0.45。Dense 略高因大多数企业问答偏向语义理解。Sparse 0.45 确保精确术语不会完全淹没。比例可调，不同场景可不同配置。

五、练习题¶

  2. 关闭 Reranker 对比：在 RetrievalPlan 中临时将 rerank 设为 False，对同一个问题跑 debug_retrieval，对比有/无 rerank 的召回排序差异。
  3. 混合检索权重实验：修改 weights 参数为 [1.0, 0.0]（纯 Dense）和 [0.0, 1.0]（纯 Sparse），观察同一问题的检索结果差异。
