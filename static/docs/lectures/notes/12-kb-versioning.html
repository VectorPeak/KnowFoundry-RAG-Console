第12讲 笔记：知识库多版本管理¶

学习目标¶

  • 能画出 KB 版本状态机的三个状态和转换条件
  • 能解释为什么版本切换是 O(1) 操作而不是批量更新 Milvus
  • 能说出每条 chunk 携带的版本相关 metadata 字段

核心知识点¶

  1. 版本状态机：STAGED（入库完成待验证）→ ACTIVE（在线检索使用）→ ARCHIVED（已下线保留）
  2. 版本切换 O(1)：激活/回滚只改本地 kb_versions.json 中的 active_version 字段，不需要批量更新 Milvus
  3. 每条 FAQ/chunk 写入 kb_version embedding_model_version chunk_schema_version 三个版本字段到 metadata，检索时过滤当前 active 版本
  4. 版本对比是激活前的安全网：用 compare_kb_versions.py 对同一批问题分别查新旧版本，对比召回差异
  5. 增量入库 + 稳定哈希 ID：未变化的文件 ID 不变，配合 IndexManifest 跳过；变化的文件新 ID 覆盖旧 ID

关键概念¶

术语 定义
STAGED 已入库但未激活的版本状态
ACTIVE 当前在线检索使用的版本
ARCHIVED 已下线但保留数据的版本，可恢复
Version Comparison 新旧版本对同批问题的召回对比，激活前的安全检查
kb_versions.json 本地版本清单文件，记录所有版本的状态和 active 指向

# qa_core/governance/kb_versions.py — 版本号生成（时间戳 + 配置哈希）
def generate_kb_version(prefix="kb", scenario_id=None):
    settings = get_settings()
    scenario = _resolve_version_scenario(scenario_id)
    stamp = utc_file_stamp()  # 20260506_103000
    config_hash = stable_hash(
        scenario.scenario_id, settings.embedding_model_version,
        settings.reranker_model_version, settings.chunk_schema_version,
        scenario.doc_collection, scenario.faq_collection,
    )[:8]
    return f"{prefix}_{scenario.scenario_id}_{stamp}_{config_hash}"
    # → "kb_enterprise_knowledge_20260506_103000_9f2a1b3c"

# qa_core/governance/kb_versions.py — 激活版本（O(1)，只改清单不改 Milvus）
def activate_version(self, kb_version):
    self.reload()
    previous = str(self.data.get("active_version") or "")
    for vid, raw in self.data["versions"].items():
        item = KnowledgeBaseVersion.from_dict(raw)
        if vid == kb_version:
            item.status = "ACTIVE"
            item.activated_at = utc_now()
        elif item.status == "ACTIVE":
            item.status = "STAGED"  # 前 active 降为 STAGED
    self.data["previous_version"] = previous
    self.data["active_version"] = kb_version
    self.save()

练习题¶

  1. 模拟版本切换：依次执行「入库新版本→评测验证→激活→提问→回滚→提问」，在每个步骤后通过状态页确认当前 active 版本和检索结果的变化。
  2. 版本对比实战：用 compare_kb_versions.py 对比两个版本对同一批问题的召回差异，观察哪些问题的召回变好了、哪些退化了。

常见面试题¶

Q1：为什么要给每个 chunk 打版本号而不是按 Collection 分版本？

按 Collection 分版本意味着每个版本一个独立 Collection，版本数量增长后存储翻倍。打版本号到 metadata 后，同一 Collection 内通过 kb_version == active 过滤即可，激活/回滚只改配置不迁数据。

Q2：激活新版本前如何保证不退化？

新版本入库后先不激活，用 compare_kb_versions.py 拿同一批评测问题分别查新旧版本，对比 Recall@K、MRR 等指标。如果RAG 回归验收不通过，拒绝激活，保留旧版本在线。
