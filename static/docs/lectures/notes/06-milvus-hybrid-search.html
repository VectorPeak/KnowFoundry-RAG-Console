第6讲 笔记：Milvus 混合检索深度解析¶

学习目标¶

  • 能解释 Milvus Hybrid Search 的 Dense + Sparse 双向量检索流程
  • 能说出过滤表达式的 5 种约束来源和安全校验机制
  • 能理解多查询变体合并去重和 Reranker 精排的完整顺序

核心知识点¶

  1. Milvus 双向量字段（dense + sparse）：dense 由 BGE-M3 生成（语义），sparse 由 Milvus BM25BuiltInFunction 服务端生成（关键词），一次入库两种向量
  2. 混合检索分数融合：默认权重 0.55(dense) + 0.45(sparse)
  3. 过滤表达式在向量检索之前执行：先用 expr 缩小搜索范围，再执行 Dense + Sparse 检索
  4. 过滤表达式包含五类约束：source / kb_version / tenant_id / dataset_id / visibility + allowed_roles
  5. search_many 对多个 query variant 分别检索 → 按 document_key 合并去重（保留最高分）→ rerank 精排
  6. langchain-milvus 是业务检索入口，PyMilvus 是底层驱动；连接别名兼容问题统一收口在 milvus_compat.py

关键概念¶

术语 定义
BM25BuiltInFunction Milvus 服务端内置函数，自动对 text 字段做中文分词 + BM25 编码
Hybrid Search Dense 向量（语义）+ Sparse 向量（关键词）加权融合检索
Filter Expression Milvus 标量过滤表达式，限制只检索符合条件的行
Consistency Level 读写一致性级别，本项目使用 Session
enable_dynamic_field 允许在不修改 Schema 的情况下写入额外 metadata 字段

# qa_core/retrieval/store.py — 混合检索核心
class MilvusHybridStore:
    def search(self, query, *, k, source_filter, kb_version,
               valid_sources, data_scope, source_type, rerank=True):
        expr = build_source_expr(source_filter, kb_version, valid_sources, data_scope)
        raw_hits = self.store.similarity_search_with_score(
            query, k=k, expr=expr,
            ranker_type="weighted",
            ranker_params={"weights": [0.55, 0.45]},  # Dense:Sparse = 55:45
        )
        hits = [RetrievalHit(document=doc, score=float(score or 0.0))
                for doc, score in raw_hits]
        if rerank and hits:
            hits = self._rerank(query, hits)
        return RetrievalResult(hits=hits, query=query, source_type=source_type, ...)

    def search_many(self, queries, *, k, ...):
        """多变体检索 → 合并去重 → rerank"""
        merged: dict[str, RetrievalHit] = {}
        for clean_query in normalize_queries(queries):
            result = self.search(clean_query, k=k, ..., rerank=False)
            merge_hits_by_document(merged, result.hits)  # 按 document_key 合并去重
        hits = sort_hits_by_score(merged.values())
        if rerank and hits:
            hits = self._rerank(queries[0], hits)
        return RetrievalResult(hits=hits[:k], ...)

# qa_core/retrieval/filters.py — 过滤表达式构建
def build_source_expr(source_filter, kb_version, valid_sources, data_scope):
    clauses = []
    if source_filter:
        if valid_sources and source_filter not in valid_sources:
            raise ValueError(f"无效的业务分类：{source_filter}")
        clauses.append(f'source == "{escape_expr_value(source_filter)}"')
    if kb_version:
        clauses.append(f'kb_version == "{escape_expr_value(kb_version)}"')
    if data_scope:
        clauses.extend(data_scope.expr_clauses())
    return " and ".join(clauses) if clauses else None

课堂踩坑小节：为什么有 milvus_compat.py¶

教学主线里，业务检索统一使用 langchain-milvus，不要求学生手写完整 PyMilvus 检索适配器。但 langchain-milvus 底层仍然依赖 PyMilvus，某些版本组合下会出现：

业务代码创建了 langchain-milvus.Milvus
  -> 内部创建 pymilvus.MilvusClient
  -> hybrid search 某些底层路径按 ORM alias 查连接
  -> pymilvus.connections 中没有对应 alias
  -> connection not found

这个问题不是 Milvus 服务端坏了，而是上层 VectorStore 和底层驱动的连接登记方式没有完全对齐。 本项目的处理不是改成纯 PyMilvus，而是把兼容修复收口到：

常见面试题¶

Q1：为什么 FAQ 和文档分为两个 Collection？

FAQ 适合高置信直出，文档适合做 LLM 生成上下文。分两个 Collection 可以独立设置 top_k、阈值和直出策略，避免混在一起时标准答案被长文档稀释，或复杂问题被相似 FAQ 误直出。

Q2：为什么用 Milvus 内置 BM25 而不是自研？

当前部署使用 Milvus 2.6.x 的 BM25BuiltInFunction 能力。自研需要维护第二套索引和入库流程。收敛到 Milvus 内意味着入库一次、检索一次、版本过滤和数据隔离同一条表达式完成。
