第17讲 笔记：LangSmith 观测与 Trace¶

学习目标¶

  • 能说出 LangSmith Trace 的核心业务 metadata 和定位 Bad Case 的排查顺序
  • 能理解轻量状态页为什么只保留状态摘要和 LangSmith 链接
  • 能解释为什么 RAG 系统的可观测性不能只靠日志

核心知识点¶

  1. 每次问答通过 langsmith_adapter.record_query_trace() 写入 LangSmith trace，metadata 至少包含：scenario_id / source / kb_version / tenant / intent / hit_type / prompt_profile / sources_count / top_source_score / stage_timings
  2. Bad Case 排查顺序：看 hit_type → 看 intent → 看 sources_count → 看 top score → 看 stage_timings → 看 prompt_profile
  3. 阶段耗时（stage_timings_ms）帮助定位性能瓶颈：intent 慢看 LLM、faq_retrieval 慢看 Milvus、llm_generation 慢看模型
  4. 状态页不复刻 LangSmith，只保留服务状态、active 版本、入库质量摘要和 LangSmith 链接
  5. Trace 写入不阻塞主链路：写入失败只记录日志，不影响用户收到答案

关键概念¶

术语 定义
LangSmith Trace 每次问答的完整调用链记录，包含 18 个业务 metadata 字段
Stage Timings 每个 Pipeline 阶段的耗时记录，由 RAGQueryContext.run_stage() 自动采集
Bad Case Review 在 LangSmith 中标注异常 trace，沉淀到 dataset 后进入回归评测

# qa_core/pipeline/runtime.py — 阶段计时的自动记录
class RAGQueryContext:
    def run_stage(self, stage_name, action):
        """执行一个阶段并自动记录耗时"""
        started = time.perf_counter()
        try:
            return action()
        finally:
            self.record_stage(stage_name, started)

    def mark_first_token(self):
        """记录首 token 到达时间"""
        if self.first_token_ms is None:
            self.first_token_ms = round((time.perf_counter() - self.started) * 1000, 2)

练习题¶

  1. 通过 LangSmith 排查问题：故意问一个明显超出当前场景范围的问题，观察 trace metadata 中的 hit_type 和 boundary_reason 是否正确记录了场景边界。
  2. 分析阶段耗时：找一个复杂问题（含多种材料的长文档答案），在 LangSmith trace 中查看各阶段耗时，判断最慢的阶段是什么并推测原因。

常见面试题¶

Q1：为什么企业路线不用项目自研追踪存储 作为主线？

本项目不再把本地追踪文件作为授课主线。企业中 Trace、Dataset、Annotation、Experiment 通常交给 LangSmith 这类成熟平台，项目代码只需要把业务 metadata 设计好。

Q2：你们怎么定位一个用户的「答得不对」？

先看 hit_type：是 faq_direct 还是 rag 还是 insufficient_context；再看 retrieval 中的 top_score 和 sources_count，如果 score 低说明召回弱，如果 score 高但 sources 不相关说明过滤表达式有问题；再看 query_variants 是否合理；最后看 prompt_profile 是否正确。
