第7讲 笔记：QAService 核心编排¶

学习目标¶

  • 能说清 QAService 三个方法（preview / stream / debug）的职责分工
  • 能理解「服务门面」模式在本项目中的具体体现
  • 能解释 HTTP 轻量预检和 WebSocket 流式主链路的分工理由

核心知识点¶

  1. QAService 是服务编排层（门面），不保存请求状态，不直接操作 HTTP/WebSocket，不实现底层检索细节
  2. 三个方法分工：preview_query 处理问候等轻量直接答案，stream_query 处理完整 RAG 流程，debug_retrieval 只跑检索半链路不调 LLM
  3. 设计意图：复杂业务问题统一走 WebSocket 流式主链路，避免 HTTP 和 WebSocket 两套实现产生不一致
  4. preview_query 返回 None 时表示「这个问题我答不了」，前端自动走 WebSocket 流式接口
  5. QAService 通过 qa_core.application.factory 做进程级缓存，每个进程只有一个实例

关键概念¶

术语 定义
服务门面 对外暴露少数几个清晰方法，内部委托给具体模块，自身保持极薄
生成器 yield 逐个产出事件，调用方可以边接收边处理，不等完整结果
请求上下文 RAGQueryContext，封装一次请求的全部临时状态
Stream Required preview_query 的特殊返回值，告诉前端必须走 WebSocket

# qa_core/application/service.py — 核心编排类
class QAService:
    def __init__(self):
        self.settings = get_settings()
        self.history = get_history_store()

    def preview_query(self, query, source_filter, session_id, ...):
        """轻量预检：问候/越界/客服直接返回，复杂问题返回 None"""
        intent = classify_intent(query, history_messages, scenario)
        if intent.direct_answer:
            self.history.add_turn(session_id, query, intent.direct_answer)
            record_query_trace(...)
            return QAResult(answer=intent.direct_answer, hit_type=intent.intent.lower(), ...)
        return None  # 前端自动走 /api/stream

    def stream_query(self, query, source_filter, session_id, ...):
        """委托 RAG Pipeline 执行完整流式问答"""
        yield from rag_stream_query(self.history, self.validate_source, ...)

    def debug_retrieval(self, query, source_filter, ...):
        """只跑检索半链路，不调 LLM"""
        return rag_debug_retrieval(self.history, self.validate_source, ...)

练习题¶

  1. 追踪 preview 到 stream 的切换：在浏览器中分别输入「你好」和「入职流程有哪些步骤」，通过 Network 面板观察前者的 HTTP 响应和后者的 WebSocket 消息。
  2. 直接调用 debug_retrieval：用 Python 脚本绕过 HTTP，直接调用 get_qa_service().debug_retrieval(...)，打印返回的 faq_sources 和 doc_sources。

常见面试题¶

Q1：为什么需要 preview_query 而不全部走 WebSocket？

问候、客服电话、越界问题不需要访问 Milvus 或 LLM。preview_query 在 HTTP 层直接返回，速度快，不消耗检索和模型资源。

Q2：QAService 为什么不能保存请求状态？

QAService 是进程级单例，多用户并发访问。如果请求状态保存在实例属性中，并发请求会互相覆盖。所有请求状态必须放在方法局部变量或 RAGQueryContext 中。
