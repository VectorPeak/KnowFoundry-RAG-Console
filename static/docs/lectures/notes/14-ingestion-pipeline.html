第14讲 笔记：文档入库与索引链路¶

学习目标¶

  • 能说出离线入库链路的 6 个步骤
  • 能解释 IndexManifest 的增量机制（指纹 → 跳过 / 更新）
  • 能理解 FAQ CSV 入库和文档入库的处理差异

核心知识点¶

  1. 离线入库 6 步：创建版本 → 构建 DataScope → 打开清单 → 加载文件 → 切分 → 写入 Milvus → 更新清单
  2. Loader 注册表模式：每种文件后缀对应一个工厂函数，新增文件格式只需一条注册项
  3. 增量入库通过 SHA256 文件指纹实现：对比 IndexManifest 中的指纹，未变化跳过，变化或新增则处理
  4. FAQ CSV 不入库为 chunk：每行 question+answer+source 作为一个完整 Document，不切分
  5. 表格文件按行转换：CSV/Excel 每行转为 content_type=table_row 的 Document，保留 sheet_name、row_number

关键概念¶

术语 定义
IndexManifest JSON 文件，记录每个已入库文件的指纹、chunk_id 列表和时间
Loader Registry 后缀→工厂函数的注册表，LOADER_REGISTRY 列表
SHA256 Fingerprint 文件内容的哈希值，任何字符变化都会导致指纹不同
normalize_documents 为 Document 补充 source、kb_version、数据隔离等 metadata

# qa_core/indexing/chunking.py — Parent-Child 双层切分（真实代码）
def split_documents(documents: list[Document]) -> tuple[list[Document], list[str]]:
    settings = get_settings()
    parent_splitter = RecursiveCharacterTextSplitter(
        chunk_size=settings.parent_chunk_size,    # 1000
        chunk_overlap=settings.parent_overlap,     # 100
        separators=CHINESE_SEPARATORS,
    )
    child_splitter = RecursiveCharacterTextSplitter(
        chunk_size=settings.child_chunk_size,      # 350
        chunk_overlap=settings.child_overlap,       # 50
        separators=CHINESE_SEPARATORS,
    )
    for doc in documents:
        content_type = str(doc.metadata.get("content_type", "")).lower()
        file_type = str(doc.metadata.get("file_type", "")).lower()

        # 表格行不切分（一行已是完整语义单元）
        if content_type.startswith("table"):
            parent_docs = [doc]
        # Markdown 先按 #/##/### 标题切，再递归切 Parent
        elif file_type == ".md":
            header_docs = MarkdownHeaderTextSplitter(
                headers_to_split_on=[("#", "h1"), ("##", "h2"), ("###", "h3")]
            ).split_text(doc.page_content)
            parent_docs = parent_splitter.split_documents(header_docs)
        else:
            parent_docs = parent_splitter.split_documents([doc])

        for parent_doc in parent_docs:
            # 稳定哈希 ID：包含 5 个版本字段 + 内容
            parent_id = stable_hash(scenario_id, kb_version,
                embedding_model_version, chunk_schema_version,
                doc_id, parent_doc.page_content)
            for child_doc in child_splitter.split_documents([parent_doc]):
                chunk_id = stable_hash(parent_id, child_doc.page_content)
                child_doc.metadata.update({
                    "parent_id": parent_id,
                    "parent_content": parent_doc.page_content,
                    "chunk_id": chunk_id,
                })

练习题¶

  1. 增量入库实战：先全量入库一个场景，然后只修改一个文档文件中的一个句子，再次执行入库脚本，观察是否只处理了变化的文件。
  2. 验证表格行处理：准备一个含 5 行数据的 CSV 文件，入库后通过 debug 接口搜索其中一行数据的关键词，确认命中的 Document 包含了 row_number 和 sheet_name。

常见面试题¶

Q1：离线入库和在线问答为什么要严格分开？

离线入库可以慢（几分钟），可以重试回滚；在线问答必须快（秒级响应）。如果把文档解析放在在线链路，用户提问时临时解析 PDF 会导致首 token 延迟增加 5-10 秒，且无法做入库质量报告。

Q2：增量入库如何避免重复处理？

每次入库前计算文件 SHA256 指纹，与 IndexManifest 记录的历史指纹对比。相同则跳过，不同或新文件则：删除旧 chunk_id → 重新切分 → 写入新 chunk → 更新 Manifest。
