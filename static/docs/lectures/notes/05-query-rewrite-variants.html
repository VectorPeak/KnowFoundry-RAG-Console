第5讲 笔记：查询改写与变体生成¶

学习目标¶

  • 能解释追问为什么需要改写以及改写的触发条件
  • 能说出 query variants 的生成策略（启发式 + LLM）
  • 能理解多轮对话中历史消息的摘要压缩机制

核心知识点¶

  1. 追问改写解决代词消解问题：用户先问「入职流程有哪些步骤」再问「那审批呢」，需要结合历史把「审批」补全为「入职审批流程」
  2. 改写触发条件：有历史消息 +（FOLLOW_UP_HINTS 命中如「那/这个/它/费用呢」或短句 ≤8 字符）
  3. Query variants 提升召回覆盖率：对同一个问题生成 2-3 个同义表达
  4. 历史摘要机制：消息超过 14 条时，将较早消息用 LLM 压缩为一段摘要，保留最近 8 条原文
  5. 变体生成有两种方式：启发式（原句 + 关键词替换）优先，LLM 补充生成更多样化的表达

关键概念¶

术语 定义
Query Rewrite 将上下文依赖的追问改写为独立检索问题
Anaphora Resolution 代词消解，将「它」「那」「这个」替换为具体指代对象
Query Variants 对同一问题生成多个同义表达，提升多路召回覆盖率
History Summary 用 LLM 将较早历史消息压缩为一段摘要文本

# qa_core/pipeline/rewrite.py — 查询改写
def rewrite_query_if_needed(query, history_messages, should_rewrite):
    if not should_rewrite or not history_messages:
        return query  # 不需要改写时直接返回原问题
    history_text = format_messages(history_messages[-8:])
    llm = get_chat_model(streaming=False)
    response = llm.invoke([
        SystemMessage(content=REWRITE_SYSTEM_PROMPT),
        HumanMessage(content=f"对话历史：\n{history_text}\n\n当前问题：{query}\n\n改写后的检索问题："),
    ])
    rewritten = str(response.content).strip()
    if not rewritten:
        raise RuntimeError("查询改写返回空结果")
    return rewritten

FOLLOW_UP_HINTS = re.compile(
    r"^(那|这个|那个|它|他们|她们|这些|上面|刚才|继续|还有|"

# 有历史 + (代词开头 或 短句≤8字符) → 标记为追问

练习题¶

  1. 模拟多轮追问：在同一 session 中连续提问：①「入职流程有哪些步骤」②「那审批呢」，通过 debug_retrieval 观察第二轮 rewritten_query 是否正确展开。
  2. 对比变体效果：对一个长查询，分别开启和关闭 use_query_variants（临时改 RetrievalPlan），对比 FAQ/doc 召回数量和 top score 的差异。

常见面试题¶

Q1：追问为什么不直接拿原问题检索？

追问通常含指代词（那/它/这个），信息不完整。如果直接拿「那审批呢」去 Milvus 检索，极短查询的向量噪音大、召回精度差。先结合历史改写为「入职审批流程需要多长时间」再检索，召回效果显著提升。

Q2：多轮对话历史越来越长怎么办？

超过 14 条消息时触发历史摘要：将较早消息用 LLM 压缩为 1200 字符以内的摘要，只保留最近 8 条完整原文。LLM 既看到全貌（摘要），又能访问最新细节（最近消息），不超 token 预算。
