四个决定的技术实现对照

业务决定核心逻辑关键函数
第一层: 查询路由先用低成本确定性规则决定 route：direct_answer 直接答，faq_exact 精确直出，retrieval 才进入检索准备。decide_route()、classify_direct_intent()、try_fast_faq_direct_answer()
第二层: 检索准备加载历史并理解用户意图——是问FAQ还是查文档? source 如何过滤? 带不带费用/合规风险? 需不需要结合历史改写? 然后制定检索计划、生成查询变体并选择 Prompt Profile。classify_intent() 检索类规则 + LLM、resolve_effective_source_filter()、rewrite_query_if_needed()、build_retrieval_plan()（4层决策）、generate_query_variants()
第三层: FAQ能直接答吗?先查FAQ库——如果命中了标准问题(如"忘记密码怎么办")且匹配度够高，直接用标准答案回复。注意: 费用/合规类问题即使FAQ命中也需要更高分数才敢直接答。search_faq()、get_faq_direct_answer()、direct_faq_answer()
第四层: 基于文档生成FAQ不够确定时查文档。从几十条候选中筛选最相关的段落(过滤低分、去重、截断过长的)，拼成上下文交给LLM。LLM生成完后还要补上"参考资料"标签。search_doc()、select_context_docs()（三重约束）、stream_llm_answer()、enforce_answer_citations()

每个阶段在做什么

0建立请求上下文 — 确定用哪个场景、哪个知识库版本、数据可见范围。不检索，只准备请求级状态。
→ create_query_context()
1查询路由 — 统一产出 direct_answer、faq_exact 或 retrieval。FAQ 精确命中是 route，不是新的 intent。
→ decide_route() → RouteDecision(route, answer, intent)
2检索准备 — 只有 route=retrieval 才进入这里，加载历史，判断用户想查什么、source 怎么过滤、问题带什么风险、追问是否要补全上下文，并生成检索计划和查询变体。
→ classify_intent() → resolve_effective_source_filter() → rewrite_query_if_needed() → build_retrieval_plan() → generate_query_variants()
3查 FAQ 库，判断能直接答吗 — 先查标准问答集。问题完全一样→无条件信任。语义相近→分数必须超过动态阈值（普通问题 0.62，费用 0.84，合规 0.86）。表格类问题只允许精确匹配（关闭相似直出）。
→ search_faq() → get_faq_direct_answer() → direct_faq_answer()
4查文档库 — FAQ 不够确定时才查。知识咨询类问题会扩大搜索范围（doc_top_k 从默认 20 扩到 complex 24）。
→ search_doc()
5筛选最相关的段落 — 从几十条候选中筛选真正进 LLM 窗口的内容。FAQ 转成"常见问题+标准答案"格式，文档优先用父块完整上下文。三重约束：条数上限 / 单条长度上限 / 总长度上限。
→ select_context_docs() → build_context()
6LLM 基于资料生成答案 — 根据问题风险类别选择提示词模板（费用走 pricing_guard、合规走 compliance_guard），把资料和问题拼成 prompt，逐 token 流式推给前端。
→ build_answer_prompt_profile() → stream_llm_answer()
7检查答案完整性 — 答案有没有标注来源？没标注的补上"参考资料"。涉及表格的确保关键字段没丢。
→ enforce_answer_citations()
8保存记录 — 问答写入 MySQL（供后续追问），摘要写入 JSONL trace（供LangSmith 分析），后台异步刷新会话摘要。
→ history.add_turn() + finish_success() + _schedule_summary_refresh()

FAQ 三条路径对比 — 业务视角

系统面对一个问题的核心判断："这个答案在 FAQ 库里吗？如果在，我能信任这个匹配吗？"答案分为三个层次。

A精确命中——"这个答案100%对"
用户问的和 FAQ 库里某条标准问题一字不差。查询路由层返回 route=faq_exact，同时标记 intent=FAQ_QUERY，直接返回标准答案。
B相似命中——"这个答案很可能对，但我不100%确定"
用户问法不同但语义相近（"怎么重置密码" vs "忘记密码怎么办"）。系统先识别意图为 FAQ_QUERY，再根据问题类型决定一个动态阈值：普通问题 0.64(默认0.72-0.08)，费用问题必须 0.84 以上。这是"信任但要验证"的路径。
C不够确定——"FAQ 给不了答案，查文档吧"
FAQ 没命中，或命中但分数不够。问题可能是知识咨询（"入职流程包含哪些步骤"），需要多段文档资料拼出完整答案。这是最慢但最全面的路径。

Milvus 混合检索 — 业务视角

检索的本质问题是："用户用这些说法提问，知识库里哪些内容最可能回答它？"解决思路分两步：先用两种方式（语义+关键词）各找一批候选，再让精排模型从中挑最好的。

1语义召回 (Dense/BGE-M3) — "入职流程"和"报到步骤"用词不同但意思相近。Embedding 模型能理解这种语义相似性。但它未必能精确匹配"HS编码8471"这种专业术语。
2关键词召回 (Sparse/BM25) — 弥补语义召回的盲区。"HS编码8471"是精确关键词，BM25 直接匹配。但它不理解"改密码"和"重置密码"是同一回事。
3双路融合 + 精排 — 两路结果按 0.55:0.45 权重合并。但向量相似度高≠真的能回答这个问题。CrossEncoder 精排模型把 query 和 passage 一起读，判断"这段内容真的在回答这个问题吗"，重新排序后取 top。

关键设计决策一览

决策原因
WebSocket 唯一在线问答入口问候、客服、越界、FAQ 和复杂 RAG 都走 /api/stream 事件链路。避免多套入口造成重复状态、重复历史写入和回答口径不一致。
规则优先 + LLM 兜底的意图识别问候、越界、客服等场景确定性强，用规则更快更稳定。LLM 只处理模糊问题，控制成本和延迟。
4 层动态检索计划不同类型问题需要不同的检索参数。费用类(threshold>=0.84)比普通 FAQ(0.62)更保守。frozen dataclass 保证创建后不可修改。
FAQ 直出分快慢两条路径快速路径是查询路由层的 route=faq_exact（首 token<500ms），正常路径是在检索计划后相似匹配（<1s）。两条路径的阈值和 rerank 策略不同。
同步生成器 + asyncio.to_threadRAG 链路（Milvus gRPC + LLM HTTP）是阻塞操作。用 asyncio.to_thread 把生成器推进放到工作线程，主线程处理 WebSocket 收发。
摘要 + 最近 8 条的历史管理不全量发送历史给 LLM（token 膨胀+注意力稀释）。早期对话压缩为摘要(<=1200字)，最近 8 轮保留原文。
TOML 驱动的多场景隔离同一套 RAG 引擎服务 8 个业务场景。每个场景独立配置 collection、source 白名单和正则匹配规则。新增场景只需复制目录改 TOML。
版本清单 JSON + Milvus expr 过滤知识库版本切换不改 Milvus 数据，只在检索 expr 中加 kb_version 过滤条件。版本激活是轻量操作。

项目目录结构

qa_core/
├── api/           # FastAPI: chat.py(WS/HTTP), dependencies.py(限流/鉴权), pages.py(静态), admin.py(管理), kb_versions.py(版本API)
├── application/   # 应用服务: service.py(QAService), factory.py(单例)
├── pipeline/      # RAG 编排: rag.py(主流程), steps.py(步骤), context.py(上下文), runtime.py(运行时)
├── retrieval/     # 检索层: store.py(Milvus), strategy.py(计划), ranking.py(排序), filters.py(过滤)
├── intent/        # 意图: classifier.py(6级规则+LLM), question_category.py(风险类别)
├── memory/        # 记忆: history.py(聊天存储), feedback.py(反馈), base.py(SQL基类)
├── governance/    # 治理: data_scope.py(数据隔离), kb_versions.py(版本管理)
├── scenarios/     # 场景: registry.py(注册表), boundary.py(边界检测)
├── prompts/       # 提示词: selector.py(选择器), profiles.py(档位), constants.py(文本)
├── llm/           # LLM: client.py(ChatOpenAI 工厂)
├── indexing/      # 入库: service.py(目录入库), faq_ingestion.py(FAQ CSV), chunking.py(切分)
├── quality/       # 质量: ingestion.py(入库报告), conflicts.py(FAQ冲突检测)
├── observability/ # 可观测: trace_writer.py(trace), traces.py(聚合), evaluations.py(评测)
├── config/        # 配置: settings.py(全局), logging_config.py, preflight.py(预检)
├── schemas.py     # Pydantic 数据模型: RetrievalDebugRequest/Response, FeedbackRequest
├── common.py      # 公共工具: utc_now(), utc_file_stamp(), read_json(), write_json(), list_json_reports()
└── utils.py       # 工具: stable_hash(), file_fingerprint(), normalize_source_from_path()
