    学习优先级与主次拆分¶

    这个项目内容已经接近企业级 RAG 教学项目。如果学生第一次学习时把所有模块都当成同等重要，会很容易迷失。正确方式是先抓住“能跑通一次高质量问答”的主链路，再逐步补治理、评测、多场景和复杂资料能力。

    一、总原则¶

    第一遍学习只回答一个问题：

    1
    用户问一个业务问题，系统如何找到正确资料，并流式生成一个有引用、可追溯的答案？

    能把这条链路讲清楚，就已经掌握了项目的主干。其他内容是为了让系统更像企业项目、更适合面试表达，但不是第一遍必须全部吃透。

    脚本学习也按同一原则处理：第一遍只掌握入库、评测、封版这些任务入口，不需要逐个阅读 scripts/ 下的所有文件。脚本的详细主次划分见 脚本使用主次指南。

    二、四层学习优先级¶

    层级 定位 是否必须 学习目标
    P0 主链路 在线问答闭环 必须掌握 能解释一次请求从页面到 RAG Pipeline 再到流式返回的全过程。
    P1 核心工程能力 检索质量、Prompt、入库、版本、评测 建议掌握 能说明为什么答案可靠、资料如何更新、如何防止质量退化。
    P2 企业化增强 多租户隔离、多场景、LangSmith Trace/Evaluation、领域指标 面试加分 能把项目讲成企业可落地方案。
    P3 扩展能力 OCR/VLM、表格复杂治理、容量评估、GraphRAG、Agent 规划 了解即可 知道边界和升级路线，不要求第一期完全掌握。

    三、P0：必须掌握的主链路¶

    P0 是教学的第一优先级，也是面试时最常被追问的部分。

    内容 对应讲义 对应代码 为什么重要
    项目启动与页面问答 第 1、12 讲 app.py、static/ 学生要先看到系统能跑，建立整体感。
    HTTP 预检 + WebSocket 流式返回 第 8、9 讲 qa_core/api/chat.py、qa_core/application/service.py 这是用户体验闭环，回答”为什么页面能流式输出”。
    意图识别 第 4 讲 qa_core/intent/classifier.py 决定后续是否检索、如何检索、用什么提示词。
    检索策略 第 5 讲 qa_core/retrieval/strategy.py 不同问题不能使用同一套 top_k 和阈值。
    查询改写与变体 第 6 讲 qa_core/pipeline/rewrite.py、qa_core/pipeline/query_variants.py 解决追问和用户表达不规范的问题。
    Milvus 混合检索 第 7 讲 qa_core/retrieval/store.py、qa_core/retrieval/filters.py RAG 的核心能力是把正确证据召回。
    上下文构建 第 9 讲 qa_core/pipeline/context.py 控制哪些证据进入 LLM，避免上下文噪声过多。
    Prompt Profile 第 10 讲 qa_core/prompts/ 不同问题类别需要不同回答口径。
    流式生成和引用增强 第 8、9 讲 qa_core/pipeline/rag.py 形成最终用户可见答案。

    P0 学完后的最低验收标准：

    1. 能画出“页面 → API → QAService → Pipeline → 检索 → LLM → 流式返回”的流程。
    2. 能解释为什么先做意图识别，再做检索计划。
    3. 能解释 Dense、Sparse、Reranker 在一次检索中的作用。
    4. 能解释为什么答案必须带来源引用。

    四、P1：建议掌握的核心工程能力¶

    P1 是让项目从“能演示”变成“能更新、能评测、能回滚”的关键。

    内容 对应讲义 对应代码 教学定位
    文档入库 第 15 讲 qa_core/indexing/ 说明知识是怎么进入 Milvus 的。
    Loader 注册表 第 15 讲 qa_core/indexing/loaders.py 展示成熟开源组件和可维护设计。
    文档切分 第 15 讲、附录 H qa_core/indexing/splitter.py 解释 chunk 粒度对召回质量的影响。
    FAQ 入库与直出 第 8、14 讲 qa_core/pipeline/steps.py、qa_core/pipeline/retrieval_steps.py 说明标准问答为什么要有快速路径。
    知识库版本 第 13 讲 qa_core/governance/kb_versions.py 更新知识库时不能直接影响线上 active 版本。
    数据隔离 第 14 讲 qa_core/governance/data_scope.py 防止跨租户、跨数据集、跨权限召回。
    RAG 回归验收 第 16 讲 scripts/evaluate_core_chain.py、eval_sets/ 用 Recall@K、MRR、关键词覆盖证明质量。
    基础测试 第 17 讲 tests/ 防止核心逻辑被改坏。

    P1 学完后的验收标准：

    1. 能从“新增一份文档”讲到“激活一个新知识库版本”。
    2. 能说明为什么知识库需要 STAGED、ACTIVE、ARCHIVED。
    3. 能说明RAG 回归验收和单元测试的区别。
    4. 能解释数据隔离如何落到 Milvus expr。

    五、P2：面试加分的企业化增强¶

    P2 不建议放在第一遍细讲。它的价值在于把项目从“RAG Demo”升级成“企业知识问答平台”。

    内容 作用 学习建议
    多场景场景包 证明同一套 RAG 能力可以复用到多个行业背景。 先理解一个场景，再看其他场景如何复用配置。
    场景边界检测 防止用户在错误场景下跨业务问答。 面试时作为“多场景隔离”的补充点。
    LangSmith Trace 排查一次回答为什么失败。 会看 trace/span 和业务 metadata 即可，不深挖自研平台。
    Bad Case 沉淀 把失败样例沉淀到 LangSmith dataset，再进入回归。 面试很加分，说明项目不是只靠主观体验。
    轻量状态页 展示当前状态、版本、入库质量和 LangSmith 链接。 学生会用即可，不必先改页面。
    企业 overlay 让样例数据更接近真实企业资料。 作为业务真实性增强，不是 RAG 主链路。
    历史报告复盘 查看入库报告、评测报告、容量报告。 用于项目交付和版本复盘。

    P2 学完后的验收标准：

    1. 能把 8 个场景解释为“同一套能力的不同业务壳”，而不是 8 套系统。
    2. 能用一个 Bad Case 说明如何定位、修复、回归。
    3. 能通过 trace 解释一次问答失败原因。

    六、P3：扩展能力，只需要知道边界¶

    P3 是二期或高级课内容，不建议压到第一期主课里。

    内容 当前定位 为什么不放进第一遍主线
    OCR 复杂图片、扫描件文字识别。 依赖重、成本高、失败率高，会分散 RAG 主线。
    VLM 理解图片、截图表格、流程图语义。 属于多模态资料治理，不是基础文本 RAG。
    复杂 Excel 解析 合并单元格、多级表头、图表、批注。 很容易变成 Office 解析专项。
    GraphRAG / 知识图谱 用实体关系增强复杂推理。 适合二期高级检索，不是一期必需。
    LangGraph Agent 规划、工具调用、多步任务。 属于二期 Agent，不应该干扰一期 RAG 闭环。
    A2A 协议 Agent 之间协作通信。 适合二期架构规划，不是当前问答主链路。
    容量评估 估算 chunk、向量、存储、耗时。 面试加分，但不影响学生先跑通系统。

    P3 的教学目标不是会实现，而是会回答：

    1
2
3
    这个能力当前为什么没有放进核心链路？
如果企业真的需要，应该如何升级？
升级后会带来什么成本和风险？

    七、18 讲主次拆分¶

    讲次 主题 优先级 第一遍学习要求
    01 项目概述与环境搭建 P0 必学，先跑通系统。
    02 RAG 核心概念 P0 必学，理解 Dense/Sparse/Reranker。
    03 LangChain 生态 P0 必学，后续所有讲次的"语言基础"。
    04 意图分类 P0 必学，后续策略都依赖它。
    05 检索策略 P0 必学，理解动态参数。
    06 查询改写与变体 P0 必学，追问补全和多路检索。
    07 Milvus 混合检索 P0 必学，RAG 召回核心。
    08 QAService 编排 P0 必学，理解服务层如何串联。
    09 RAG Pipeline P0 必学，全项目主流程。
    10 Prompt Profile P0 必学，控制答案结构和业务边界。
    11 FastAPI 与异步 P1 学完 P0 主链路后学习，理解 Web 服务骨架。
    12 应用入口与前置校验 P1 理解为什么环境必须完整。
    13 知识库版本 P1 建议掌握，体现工程可靠性。
    14 数据隔离 P1 建议掌握，企业项目必问。
    15 文档入库 P1 建议掌握，解释知识如何进入系统。
    16 RAG 回归验收与入库质量 P1 建议掌握，证明效果不是拍脑袋。
    17 测试与接口验收 P1 先掌握核心测试思路。
    18 LangSmith 观测与 Trace P2 第二遍学习，面试加分。

    八、推荐教学节奏¶

    1. 快速体验课：1 天¶

    目标：学生知道项目能做什么。

    学习内容：

    1. 启动项目。
    2. 选择一个场景提问。
    3. 看流式输出、引用来源、状态页状态。
    4. 讲清 RAG 和普通 ChatGPT 的区别。

    对应内容：第 1 讲 + 第 2 讲前半。

    2. 主链路课：3 到 5 天¶

    目标：学生能解释一次问答主流程。

    学习内容：

    1. 意图识别。
    2. 检索计划。
    3. 查询改写。
    4. Milvus 混合检索。
    5. QAService 和 RAG Pipeline。
    6. Prompt Profile。

    对应内容：第 3 到 9 讲。

    3. 工程化课：2 到 3 天¶

    目标：学生能解释如何更新知识库、如何保证质量。

    学习内容：

    1. 应用入口与前置校验。
    2. LangChain 生态。
    3. 知识库版本。
    4. 数据隔离。
    5. 文档入库。
    6. RAG 回归验收。
    7. 基础测试。
    8. FastAPI 异步框架 + 应用入口。

    对应内容：第 11 到 18 讲。

    4. 面试拔高课：1 到 2 天¶

    目标：学生能把项目讲成企业级方案。

    学习内容：

    1. 多场景复用。
    2. LangSmith Trace 诊断。
    3. Bad Case 沉淀。
    4. 企业 overlay。
    5. 二期 Agent / GraphRAG / 多模态规划。

    对应内容：第 18 讲 + 面试材料 + 规划文档。

    九、第一遍可以暂时跳过的内容¶

    这些内容不是没价值，而是不适合第一遍深挖：

    • 复杂 OCR / VLM 细节；
    • 复杂 Excel 语义还原；
    • 所有 scripts 脚本的逐行解释；
    • 所有 8 个业务场景的完整资料细节；
    • Mermaid 架构图里的每个辅助节点；
    • 容量评估和历史报告复盘的实现细节；
    • 二期 Agent、LangGraph、A2A、GraphRAG 设计细节；
    • 状态页前端样式和非核心 UI 交互。

    第一遍只需要知道它们存在，以及它们服务于哪个工程目标。

    十、最小闭环作业¶

    学生完成下面 5 件事，就说明已经掌握一期主线：

    1. 启动项目并完成一次流式问答。
    2. 解释该问题的意图分类结果。
    3. 解释该问题的检索计划为什么这样设置。
    4. 找到至少一条引用来源，并说明它是怎么从 Milvus 召回来的。
    5. 修改一份文档或 FAQ，重新入库并让新版本生效。

    扩展作业可以放到第二轮：

    1. 制作一个 Bad Case 并加入评测集。
    2. 新增一个表格 CSV，验证 table_row 检索。
    3. 用 trace 分析一次回答不理想的原因。
    4. 选择一个业务场景，包装成简历项目描述。

    十一、面试表达主线¶

    面试时不要从“项目有很多功能”开始讲，而要按主次讲：

    1
2
3
4
5
6
    这个项目的核心是一套企业级 RAG 问答主链路。
在线侧通过 HTTP 预检和 WebSocket 实现流式问答；
检索侧通过意图识别、动态检索计划、查询改写、Milvus Dense+Sparse 混合检索和 Reranker 提高召回质量；
生成侧通过 Prompt Profile、引用增强和上下文筛选保证答案可控；
工程侧通过知识库版本、数据隔离、入库质量检查、领域评测指标和 LangSmith Trace/Evaluation 保证系统可维护、可回滚、可诊断。
OCR/VLM、GraphRAG、LangGraph Agent 属于后续扩展方向，不混入一期主链路。

    这条表达主线能让面试官先听到项目主干，再听到企业化能力，最后听到二期规划。主次清楚，项目就不会显得堆功能。
