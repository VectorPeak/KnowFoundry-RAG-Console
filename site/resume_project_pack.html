    多场景 RAG 项目简历包装材料¶

    本文档用于把当前项目转换成可写简历、可面试讲解的项目表达。核心原则是：不要把同一套代码简单换行业名称，而要讲清楚每个业务场景的资料结构、风险边界和工程能力。

    1. 包装原则¶

    简历中不要写成“做了一个 AI 问答系统”。这个说法太泛，和普通 Demo 区分不开。

    推荐表达为：

    • 我做的是企业级 RAG 知识问答平台；
    • 有明确业务资料来源、source 分类、知识库版本和入库质量检查；
    • 使用 Milvus Hybrid Search，不是只做向量相似度；
    • 支持 FAQ 标准答案和文档 RAG 两条路径；
    • 对费用、合规、安全、赔付等高风险问题有 Prompt Profile 控制；
    • 有评测指标证明效果，而不是只靠页面演示。

    2. 推荐主推项目¶

    2.1 跨境贸易风控 RAG 知识问答平台¶

    适合人群：想突出供应链、跨境电商、贸易合规、风控系统背景的学生。

    项目背景：

    企业在跨境贸易中需要处理海关申报、HS 编码、制裁筛查、信用证、物流责任和单证一致性。传统知识库靠人工搜索文档，响应慢且容易遗漏风险口径。

    简历描述：

    基于 LangChain + Milvus Hybrid Search 构建跨境贸易风控 RAG 知识问答平台，覆盖海关申报、制裁筛查、信用证审单、贸易术语和单证一致性场景。系统支持 FAQ 标准答案直出、文档混合检索、知识库多版本、source 自动推断、合规类 Prompt 路由和来源引用，通过 Recall@K、MRR 和 Prompt 命中率做回归验收。

    核心职责：

    • 设计 customs/sanction/payment/logistics/documentation 五类 source；
    • 使用 Milvus dense + BM25 sparse 完成混合召回；
    • 对 HS 编码争议、最终用途不清、信用证软条款等问题进入合规或费用强口径模板；
    • 建立业务深度评测样本，验证召回、来源隔离和 Prompt 路由；
    • 用知识库版本控制新增资料上线，避免直接覆盖旧版本。

    可讲难点：

    • HS 编码、信用证、单证一致性不是普通 FAQ，需要回答“可以做什么”和“不能直接承诺什么”；
    • 同一问题可能同时命中 FAQ 和正文，需要 FAQ 标准口径优先，同时保留文档依据；
    • 跨境风险类问题不能让模型自由发挥，所以用 Prompt Profile 强制区分已确认和未确认。

    面试展开点：

    • 为什么不用 RedisSearch；
    • 为什么 Milvus 2.6.x 内置 BM25 足够；
    • 如何处理知识库版本回滚；
    • 如何防止不同场景串库；
    • 如何证明制裁、报关、信用证类问题走了合规模板。

    2.2 工程项目资料与施工规范 RAG 问答助手¶

    适合人群：想突出工程项目、资料管理、标准规范、多版本文档检索背景的学生。

    项目背景：

    工程项目中存在图纸版本、施工规范、进度计划、质量验收和安全资料。资料分散且版本复杂，现场人员容易引用旧图纸或缺少验收依据。

    简历描述：

    构建工程项目资料与施工规范 RAG 问答助手，面向设计图纸、施工规范、进度资料、质量验收和安全资料提供多文档检索与风险口径回答。系统支持图纸版本追溯、规范冲突处理、隐蔽工程验收资料查询和安全交底资料核查，并通过知识库版本、source 过滤和合规 Prompt 控制回答边界。

    核心职责：

    • 设计 drawing/specification/schedule/quality/safety 五类工程资料 source；
    • 补充图纸会审、强制性条文、关键线路延期、检验批不一致、安全交底等样本；
    • 对“图纸和规范冲突”“安全技术交底只有口头说明”等问题进入合规模板；
    • 用 parent-child chunk 保留工程资料上下文，避免只召回孤立短句；
    • 在状态页展示来源文件、版本、模板和检索耗时。

    可讲难点：

    • 工程资料天然适合讲多文档、多版本和标准规范检索；
    • 有些问题不能直接回答“可以施工”，必须说明依据不足或需要负责人确认；
    • 质量验收和安全资料属于高风险问题，必须比普通知识问答更保守。

    2.3 招投标合规与合同履约 RAG 风控平台¶

    适合人群：想突出 ToB 项目管理、招采合规、合同履约、项目交付背景的学生。

    项目背景：

    招投标和合同履约中常见风险包括投标材料缺失、非标准付款条款、交付范围变化、口头验收、需求频繁变更。普通问答系统容易把风险问题答成流程说明，缺少口径控制。

    简历描述：

    面向招投标和合同履约场景构建 RAG 风控平台，覆盖投标文件、合同条款、交付履约、验收管理和风险预警。系统通过 FAQ + 文档混合检索、Prompt Profile 和知识库入库质量检查，控制付款承诺、验收结论和履约责任等高风险回答边界。

    核心职责：

    • 设计 bidding/contract/delivery/acceptance/risk 五类业务 source；
    • 对投标保证金、合同变更、交付范围蔓延、口头验收、需求变更建立 FAQ 和正文资料；
    • 将保证金、回款等问题路由到 pricing_guard；
    • 将合同变更、履约责任等问题路由到 compliance_guard；
    • 用评测集验证命中来源和模板路由准确性。

    2.4 保险理赔材料审核与 RAG 知识问答助手¶

    适合人群：想突出金融保险、材料审核、风控审核、规则解释背景的学生。

    项目背景：

    保险理赔涉及保单条款、理赔材料、责任认定、除外责任和赔付结算。系统不能提前承诺赔付金额，也不能在材料不足时给出确定结论。

    简历描述：

    构建保险理赔材料审核与 RAG 知识问答助手，支持保单条款解释、理赔材料清单、责任认定、除外责任和赔付结算问答。系统通过知识库版本、来源引用、合规 Prompt 和费用 Prompt 控制“是否赔付、赔付金额、打款账户”等敏感口径。

    核心职责：

    • 设计 policy/claim_material/liability/exclusion/settlement 五类 source；
    • 对保单复效、影像材料不清、责任争议、既往症、账户一致性建立样本；
    • 对既往症和责任争议使用合规模板；
    • 对收款账户和赔付结算使用费用强口径模板；
    • 用业务深度回归评测验证召回和模板路由。

    3. 可选项目包装¶

    企业内部知识库智能问答平台¶

    适合表达企业制度、HR、IT、财务内部服务台能力。亮点是多 source、FAQ 直出、预算审批和权限回收等内部流程问答。

    简历句子：

    基于 LangChain 和 Milvus Hybrid Search 搭建企业内部知识库智能问答平台，覆盖 HR 制度、IT 支持和财务报销，支持 FAQ 直出、文档 RAG、权限 source 过滤、知识库多版本和流式输出。

    SaaS 客服知识库智能助手¶

    适合表达客服降本、产品文档问答、账号计费集成支持。亮点是退款承诺、API 限流排查和标准 FAQ。

    简历句子：

    构建 SaaS 客服知识库智能助手，覆盖账号登录、计费发票和开放集成，支持 FAQ 标准答案直出、API 排障步骤输出、退款类强口径模板和多场景评测回归。

    制造业设备运维知识助手¶

    适合表达设备 SOP、巡检、告警、安全规范。亮点是排障模板和安全作业合规模板。

    简历句子：

    构建设备运维知识助手，覆盖巡检标准、温度告警和安全规范，通过 Milvus Hybrid Search 和 rerank 召回 SOP 文档，对反复告警输出排查步骤，对受限空间作业输出安全确认边界。

    企业合规制度智能问答系统¶

    适合表达法务合规、审计、隐私保护。亮点是合规模板、客户数据导出、供应商尽调和审计闭环。

    简历句子：

    构建企业合规制度智能问答系统，覆盖合同审查、审计整改和隐私保护，对供应商尽调、客户数据导出等问题使用合规 Prompt 控制已确认和未确认边界。

    5. 面试时要避免的说法¶

    不要说：

    • “我做了一个 ChatGPT 套壳。”
    • “我把文档向量化后查出来给模型。”
    • “BM25 是我手写的。”
    • “MySQL 存知识库，Milvus 可有可无。”
    • “缓存一下就能解决性能。”
    • “什么业务都能接。”

    推荐说：

    • “这是一个多场景 RAG 主链路项目，场景扩展靠配置，主链路只有一套。”
    • “知识检索统一走 Milvus Hybrid Search，MySQL 只负责历史和反馈。”
    • “FAQ 和文档分集合，是为了区分标准口径和解释依据。”
    • “费用、合规、安全问题用 Prompt Profile 做强约束。”
    • “我用评测集和入库质量检查证明改动没有让召回、模板路由和场景隔离退化。”

    6. 项目边界¶

    一期已经完成 RAG 闭环，不建议继续堆更多行业场景。下一阶段如果要升级，应进入 Agent：

    • 用 LangGraph 编排工具调用；
    • 将风控问答升级为风险处置流程；
    • 将工程资料问答升级为资料核查任务；
    • 将 SaaS 客服问答升级为工单草稿生成；
    • 将保险理赔问答升级为材料缺失检查流程。

    简历中可以写“一期 RAG 已完成，二期规划 Agent 工作流”，不要把未实现的 Agent 能力写成已上线功能。
