    一期 RAG 快速复盘¶

    这份文档用于演示、交付和面试前的快速检查。它不替代详细架构文档，只回答一个问题：当前一期 RAG 是否已经形成可运行、可评测、可回滚、可解释的核心闭环。

    1. 一期边界¶

    一期只做 RAG 核心能力：

    • 多场景知识库问答；
    • FAQ 标准答案直出；
    • 文档 Hybrid Search 检索；
    • source 和场景隔离；
    • Prompt Profile 路由；
    • 知识库多版本；
    • 流式输出；
    • 聊天历史；
    • 入库质量检查、RAG 回归验收、性能基线和 LangSmith 观测。

    二期再做 Agent。审批流、工具调用、任务编排、多角色协同不混入一期主问答链路，避免学生还没理解 RAG 主链路，就被 Agent 概念拉散。

    2. 一期是否就绪¶

    最终结论看独立质量检查是否通过。

    它汇总以下关键证据：

    证据 说明
    表格行召回验收 验证 CSV/Excel 这类结构化资料不会被普通文本切分破坏。
    负样本边界验收 验证场景越界、source 选错、证据不足时不会硬答。
    全场景版本对比 验证新旧知识库版本切换前没有召回退化。
    性能回归验收 验证首 token、总耗时和阶段耗时仍在可接受范围。
    API 合同冒烟 验证管理接口、版本接口、评测趋势、bad case 闭环可用。
    页面/流式冒烟 验证问答页、状态页和 WebSocket 流式事件可用。

    推荐按主题执行核心检查：

    1
2
3
4
    python scripts/check_project_guardrails.py
python scripts/check_evaluation_gate.py --dataset eval_sets/business_depth_regression.json --limit 40
python scripts/api_e2e_smoke.py --base-url http://127.0.0.1:8000

    这样每一步都能对应到课堂里的一个质量主题，不再需要额外讲总编排脚本。

    3. 状态页怎么看¶

    访问：

    1
    http://127.0.0.1:8000/admin

    重点看三个区域：

    • 一期就绪总览：直接看当前一期是否可交付；
    • 全场景版本对比：看每个场景的新旧知识库召回是否退化；
    • 评测报告 / 性能基线 / Bad Case 闭环：看质量、耗时和问题样本是否有闭环。

    状态页只读取已经生成的 JSON 报告，不会因为刷新页面重新调用模型或 Milvus 检索。这一点适合在面试里解释：状态页是“观察层”，不是“执行层”。

    4. 面试讲法¶

    可以按这条线讲：

    1. 先讲主链路：用户问题进入 QAService，完成场景解析、意图识别、source 推断、FAQ 检索、文档检索、重排、Prompt 路由和流式生成。
    2. 再讲治理能力：入库质量报告、低质量 chunk 检测、FAQ/正文冲突、OCR 风险、表格专用切分。
    3. 再讲版本能力：每次重建都会生成新 kb_version，激活前用全场景版本对比确认没有召回退化。
    4. 最后讲验收能力：用独立脚本证明单测、质量、评测、性能和接口冒烟都能通过。

    一句话总结：

    这个项目不是只实现了一个能聊天的 RAG，而是把企业知识库从资料入库、版本发布、检索问答、RAG 回归验收到线上诊断做成了一套闭环。
